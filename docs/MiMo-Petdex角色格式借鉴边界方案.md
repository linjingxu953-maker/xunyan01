# MiMo 执行参考：Petdex 角色格式借鉴边界方案

> 目的：给 MiMo 作为执行约束，避免把 Petdex 的角色格式、状态枚举、精灵图规格直接照搬进寻研项目。

## 一句话结论

寻研已有角色“妍”和现有角色配置体系。后续角色格式应以项目内的角色配置为主，Petdex 只作为外部角色资产组织方式的参考，不作为主模型、不作为强制规格、不作为状态机命名来源。

## 当前主格式

寻研的角色主格式应围绕当前 UI 层的角色配置继续扩展，核心对象是角色资料、人格设定、状态图片、颜色、头像、语言风格、回复偏好和自定义提示词。

主格式必须支持：

| 分类 | 说明 |
|------|------|
| 基本信息 | 角色名称、角色描述、头像文字、角色定位 |
| 人格设定 | 性格、语气、语言风格、回复长度、口头禅、是否使用 emoji |
| 外观资源 | 默认头像、图片目录、各任务状态对应图片 |
| 状态表达 | idle、listening、working、waiting、completed、error 等项目内状态 |
| 扩展字段 | 作者、版本、许可证、标签、兼容信息 |

当前角色“妍”应作为默认角色样例保留，不能因为引入 Petdex 参考而重命名、替换或迁移成 Petdex 宠物格式。

## Petdex 只借鉴什么

可以借鉴以下设计思想：

| 可借鉴点 | 我们如何吸收 |
|----------|--------------|
| 角色包清单 | 可以有一个角色包 manifest，用来描述角色名称、资源、状态图、版本和作者 |
| 资源完整性检查 | 加载角色时检查图片文件是否存在，缺失时使用默认图或提示用户 |
| 动画元数据 | 对精灵图、Lottie、状态帧率等信息做描述，但必须是可选能力 |
| 状态到动画映射 | 可以参考“任务状态映射动画状态”的思想，但状态名以寻研项目内状态为准 |
| 社区角色包思路 | 后续角色市场可以参考标准化提交和审核流程 |

## 明确不能照抄什么

MiMo 不应执行以下操作：

| 禁止项 | 原因 |
|--------|------|
| 不要把 Petdex 的 pet.json 作为项目唯一角色格式 | 寻研角色包含人格、语气、任务状态和 UI 交互，不是纯宠物装饰 |
| 不要强制 8 行 9 列、192×208、72 帧精灵图 | 这会限制用户直接放入普通 PNG/JPG/WebP 状态图 |
| 不要直接复用 idle/wave/run/failed/review/jump 作为主状态枚举 | 项目已有更细的状态，如 listening、understanding、planning、waitingApproval、memoryConfirm |
| 不要把动画播放器作为当前主链路前置条件 | 当前优先稳定角色图片、状态切换和设置入口，动画是后续增强 |
| 不要改写 UI 层的角色主模型 | UI 层由 Codex 维护，MiMo 如需对接应通过接口或文档先对齐 |
| 不要导入 Petdex 源码 | 只参考结构和思路，避免引入不必要许可、技术栈和维护风险 |

## 建议的寻研角色包结构

角色包可以采用“项目主 manifest + 可选兼容信息”的结构。

必需内容：

| 字段组 | 内容 |
|--------|------|
| schema | 固定为寻研自己的角色格式版本，例如 xunyan.character.v1 |
| profile | 角色名称、描述、角色定位、人格设定 |
| appearance | 头像、图片目录、状态图片映射、主题色 |
| states | 项目内状态名和显示名称 |
| metadata | 作者、版本、许可证、标签 |

可选内容：

| 字段组 | 内容 |
|--------|------|
| animation | 是否使用动画、动画类型、帧率、循环策略 |
| petdexCompatibility | 是否包含 Petdex 风格精灵图、网格信息、状态行映射 |
| market | 未来角色市场需要的预览图、评分、下载地址等 |

这里的 petdexCompatibility 只能是兼容段，不能反向决定主格式。

## 状态映射原则

以寻研项目内状态为主，Petdex 状态只作为导入外部精灵图时的临时映射。

| 寻研状态 | 说明 | 可选 Petdex 映射 |
|----------|------|------------------|
| Idle | 待机 | idle |
| Listening | 用户输入或语音监听 | idle 或 wave |
| Understanding | 理解问题、读取上下文 | review |
| Planning | 规划任务 | review |
| Working | 执行任务 | run |
| WaitingApproval | 等待权限确认 | review |
| MemoryConfirm | 等待记忆确认 | review |
| Completed | 完成反馈 | wave |
| Error | 错误反馈 | failed |

如果外部角色包缺少某个状态图，应该回退到 Idle 或默认角色图，而不是要求用户补齐 Petdex 全套状态。

## MiMo 可做的范围

如果 MiMo 接下来处理角色格式，只建议做以下非 UI 主链路工作：

| 工作项 | 说明 |
|--------|------|
| 角色包 schema 文档 | 描述寻研自己的角色包字段和版本策略 |
| 兼容性校验规则 | 检查角色包资源是否齐全、字段是否合理 |
| 状态映射表 | 提供外部状态到寻研状态的映射建议 |
| 导入转换策略 | 把 Petdex 风格资源转换成寻研角色包，而不是让寻研变成 Petdex |
| 单元测试 | 测试 manifest 解析、缺图回退、状态映射，不涉及 UI XAML |

MiMo 不应直接修改：

- `src/DesktopMascot.UI/Views`
- `src/DesktopMascot.UI/ViewModels`
- 当前角色“妍”的默认视觉呈现
- 设置中心的 UI 结构

## Codex 继续负责的范围

Codex 继续负责：

| 工作项 | 说明 |
|--------|------|
| 角色外观页 | 角色资料、状态图预览、图片目录、角色包导出 |
| 对话框内设置页 | 保持设置与对话框一体化 |
| 状态图实时切换 | 根据 UI 状态切换角色图片 |
| 动画入口 | 后续在 UI 层逐步加轻量动画，不影响现有图片模式 |

## 执行优先级

1. 先固定寻研自己的角色包 manifest。
2. 再做当前角色“妍”的导出和预览。
3. 再做资源完整性检查和缺图提示。
4. 最后再考虑 Petdex 精灵图导入转换。

不要把 Petdex 动画播放器、角色市场、提交流程放到当前主线任务里。

## 给 MiMo 的执行口径

请按“寻研主格式 + Petdex 可选兼容层”的方向处理，不要直接复用 Petdex 的角色模型。当前项目已有角色“妍”和角色设置 UI，Petdex 只能作为角色包组织、资源检查和动画元数据的参考。任何涉及 UI 层 View、ViewModel、角色外观页、设置页的修改都先交给 Codex 或等待接口对齐。
