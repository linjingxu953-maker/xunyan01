# 桌面 AI 互动小人设计方案

生成日期：2026-06-10  


## 1. 项目定位

本项目不是一个普通聊天窗口，也不是偏娱乐的桌宠，而是一款面向 Windows 桌面的常驻式 AI 工作助手。它以 2D 小人的形式出现在桌面上，能够理解当前屏幕、浏览器页面和用户授权的本地文件，接收用户任务，进入多步骤 Agent 工作流，并通过动画状态让用户看清楚 AI 正在做什么。

一句话定位：

> 一款 Windows 桌面常驻 AI 小人，具备屏幕上下文理解、文件和浏览器读取、任务工作流执行、手动确认式长期记忆和高危操作确认能力。

核心价值：

1. 用户看到的是一个可交互的小人，而不是传统聊天框。
2. 用户可以直接基于当前屏幕发起任务，减少复制、粘贴和手动整理。
3. AI 能执行多步骤任务，而不是只给一句回答。
4. AI 执行过程通过状态动画、进度和工具调用记录可视化。
5. 文件写入、命令执行、记忆保存等敏感行为必须经过用户确认。
6. 用户能够控制 AI 记住什么、忘记什么、能访问哪些目录。

## 2. 初版范围

### 2.1 平台范围

初版只做：

| 范围 | 决策 |
|---|---|
| 操作系统 | Windows 10 / Windows 11 |
| 运行形态 | 桌面常驻悬浮小人 |
| 安装方式 | Windows 安装包 |
| 后台能力 | 托盘、快捷键、开机自启、本地数据目录 |

初版不做 macOS、Linux、手机端、独立浏览器插件、微信/QQ/飞书机器人。原因是 Windows 桌面权限、悬浮窗、文件访问、快捷键、托盘、安装包和开机自启已经足够复杂，初版应先把单平台体验打透。

### 2.2 交互范围

初版支持：

1. 桌面悬浮小人。
2. 文本输入。
3. 快捷键唤起。
4. 鼠标点击小人。
5. 任务状态动画。
6. 气泡式反馈。
7. 简易任务详情面板。
8. 权限确认弹窗。
9. 记忆确认弹窗。
10. 设置页与托盘菜单。

初版不做：

1. 语音输入和语音播报。
2. 3D 小人。
3. 高复杂度 Live2D 骨骼动画。
4. 换装系统、多角色系统、表情商店。
5. 完全自主操作电脑。

动画技术优先级：

| 阶段 | 技术 |
|---|---|
| 初版优先 | Lottie |
| 初版备用 | PNG 序列帧 |
| 后期增强 | Live2D / Spine |

初版动画目标不是精美，而是状态清晰。

### 2.3 能力范围

初版支持：

1. 读取当前屏幕内容。
2. 读取当前窗口标题和应用名。
3. 读取选中文本。
4. 读取浏览器当前页面内容。
5. 读取用户授权的本地文件。
6. 生成新文件。
7. 调用 Agent Runtime 执行多步骤任务。
8. 长期记忆，但必须手动确认。
9. 文件写入和命令执行前必须确认。
10. 记录任务历史、本地日志和工具调用。

初版禁止：

1. 自动删除文件。
2. 自动提交网页表单。
3. 自动支付。
4. 自动发送邮件。
5. 自动发布内容。
6. 自动操作账号。
7. 无确认执行终端命令。
8. 无确认修改用户文件。

## 3. 目标用户与核心场景

### 3.1 目标用户

首批目标用户建议聚焦三类：

1. 开发者：需要分析报错、理解项目目录、生成脚本和文档。
2. 知识工作者：需要总结网页、整理资料、生成报告。
3. AI 重度用户：已经习惯使用 AI，但希望减少复制粘贴和上下文准备成本。

### 3.2 核心场景

场景一：当前网页总结

1. 用户打开网页。
2. 按快捷键唤起小人。
3. 输入“总结这个页面”。
4. 系统读取当前浏览器标题、URL 和页面正文。
5. Agent 生成摘要、要点和后续建议。
6. 小人从 ReadingContext 切换到 Working，再进入 Completed。

场景二：当前报错分析

1. 用户在 IDE 或终端中遇到报错。
2. 选中报错文本或直接唤起小人。
3. 输入“帮我看这个报错”。
4. 系统读取当前窗口、选中文本和必要上下文。
5. Agent 解释原因、给出修复步骤。
6. 若要修改文件，弹出权限确认。

场景三：项目目录诊断

1. 用户授权一个项目目录。
2. 输入“检查这个项目结构并给我改进建议”。
3. 系统只读取授权目录内文件。
4. Agent 生成结构分析、风险点和建议。
5. 若需要生成报告文件，必须二次确认写入路径。

## 4. 产品形态与页面设计

### 4.1 桌面主界面

桌面主界面由小人主体、气泡反馈、输入框和任务状态入口组成。默认位置在桌面右下角，用户可拖动、缩放、隐藏。

结构：

```text
桌面小人
  ├── 小人动画区
  ├── 简短气泡反馈
  ├── 输入框
  ├── 当前任务状态
  └── 展开按钮
        └── 任务详情面板
```

主界面状态：

| 状态 | 表现 |
|---|---|
| 空闲 | 轻微呼吸、眨眼 |
| 唤起 | 看向用户，输入框展开 |
| 思考 | 思考表情或整理清单 |
| 执行 | 忙碌动画 |
| 等待确认 | 举手提示，弹出确认卡片 |
| 完成 | 点头或递交结果 |
| 失败 | 困惑状态，并给出可重试入口 |

### 4.2 任务详情面板

任务详情面板用于解决 Agent 黑箱问题，必须让用户知道当前任务进展。

包含：

1. 任务标题。
2. 当前状态。
3. 执行步骤。
4. 进度条。
5. 正在调用的工具。
6. 工具调用记录。
7. 等待用户确认的操作。
8. 最终结果。
9. 保存结果按钮。
10. 中断任务按钮。

### 4.3 设置页

设置页包含：

1. 模型设置。
2. API Key 设置。
3. 权限设置。
4. 记忆设置。
5. 数据目录。
6. 开机自启。
7. 快捷键设置。
8. 日志导出。
9. 关于产品。

### 4.4 记忆中心

记忆中心包含：

1. 用户记忆。
2. 项目记忆。
3. 技能记忆。
4. 任务历史。
5. 搜索记忆。
6. 编辑记忆。
7. 删除记忆。
8. 导出记忆。

### 4.5 权限中心

权限中心包含：

1. 已授权目录。
2. 屏幕读取权限。
3. 浏览器读取权限。
4. 文件读取权限。
5. 命令执行权限。
6. 本次任务临时授权。
7. 永久授权。
8. 撤销授权。
9. 权限操作日志。

## 5. 系统架构

### 5.1 总体架构

```text
Windows Desktop Shell
  ├── Floating Mascot UI
  ├── Animation State Machine
  ├── Input Bubble
  ├── Task Panel
  └── Tray / Settings

Task Router
  ├── classify task
  ├── build context request
  ├── call agent runtime
  └── stream task events

Context Adapter
  ├── screen context
  ├── active window context
  ├── browser context
  ├── selected text context
  └── local file context

Agent Runtime
  ├── planning
  ├── tool calling
  ├── result generation
  └── memory proposal

Permission Manager
  ├── file read/write approval
  ├── command approval
  ├── memory approval
  └── audit log

Local Data Layer
  ├── SQLite
  ├── Markdown memory files
  ├── config
  ├── logs
  └── cache
```

### 5.2 模块职责

| 模块 | 职责 |
|---|---|
| Desktop Shell | 悬浮窗、小人动画、输入框、气泡、托盘、设置 |
| Animation State Machine | 把任务事件映射成小人状态和动画 |
| Task Router | 判断任务类型、组装上下文、分发给 Agent |
| Context Adapter | 读取屏幕、窗口、浏览器、选中文本和授权文件 |
| Agent Runtime | 规划任务、调用工具、生成结果和记忆建议 |
| Permission Manager | 管理授权、确认弹窗、风险拦截和审计日志 |
| Memory Center | 管理用户记忆、项目记忆、技能记忆和任务历史 |
| Local Data Layer | 存储配置、任务、事件、工具调用、权限和日志 |

## 6. 技术选型

### 6.1 桌面端

推荐优先方案：

```text
C# / Avalonia
```

理由：

1. Windows 桌面能力成熟。
2. 适合做悬浮窗、托盘、设置页、全局快捷键。
3. 资源占用相对可控。
4. 后期有跨平台可能。

备选方案：

```text
Tauri + Rust + 前端动画
```

不建议初版使用 Electron，主要原因是体积和内存占用偏高，而桌面小人需要常驻运行，资源占用会直接影响用户感知。

### 6.2 Agent 层

推荐方向：

```text
Hermes Agent Runtime
```

职责：

1. 接收 Task Router 发来的任务。
2. 使用上下文执行规划。
3. 调用工具。
4. 返回状态事件流。
5. 生成最终结果。
6. 提出可保存的记忆建议。

需要注意：对话中提到 Everywhere、Hermes Agent、OpenHuman 等项目。商业产品实现时不能直接复制第三方代码进入产品，需要逐项复核许可证、依赖协议和可商用边界。

### 6.3 上下文感知层

初版能力：

1. 当前窗口标题。
2. 当前应用名。
3. 屏幕截图或 OCR 结果。
4. 选中文本。
5. 浏览器标题、URL、正文。
6. 用户授权目录下的文件内容。

上下文读取原则：

1. 最小读取，只读取完成任务所需内容。
2. 明示来源，让用户知道读取了什么。
3. 授权目录外禁止读取。
4. 敏感内容尽量本地处理。
5. 高风险操作前必须确认。

### 6.4 本地存储

推荐：

```text
SQLite + Markdown / JSON
```

用途：

| 数据 | 存储 |
|---|---|
| 任务记录 | SQLite |
| 状态事件 | SQLite |
| 工具调用 | SQLite |
| 权限记录 | SQLite |
| 用户记忆 | Markdown / JSON |
| 项目记忆 | Markdown / JSON |
| 配置 | JSON |
| 日志 | text / jsonl |

正式安装后的本地数据目录建议：

```text
C:/Users/{username}/AppData/Roaming/DesktopAIMascot/
  memory/
  db/
  logs/
  config/
  cache/
```

## 7. 状态机设计

状态机是本项目的核心。每一个 Agent 动作都必须映射到小人状态，否则小人只会变成聊天框皮肤。

### 7.1 初版状态

| 状态 | 含义 | 动画 |
|---|---|---|
| Idle | 空闲 | 呼吸、眨眼 |
| Listening | 接收任务 | 看向用户、轻微前倾 |
| Understanding | 理解用户意图 | 思考表情 |
| ReadingContext | 读取屏幕、文件或网页 | 看屏幕、翻资料 |
| Planning | 规划步骤 | 展开清单、点头 |
| WaitingApproval | 等待用户确认 | 举手、提示 |
| Working | 执行任务 | 敲键盘、忙碌 |
| MemoryConfirm | 请求保存记忆 | 拿出笔记本 |
| Reporting | 生成汇报 | 整理文件 |
| Completed | 完成 | 点头、递交结果 |
| Error | 失败 | 困惑、提示重试 |

### 7.2 状态事件协议

```json
{
  "task_id": "task_20260610_001",
  "state": "Working",
  "message": "正在读取当前页面内容",
  "progress": 35,
  "tool": "browser_context",
  "need_user_action": false
}
```

### 7.3 需要确认的事件

必须确认：

1. 写入文件。
2. 覆盖文件。
3. 执行命令。
4. 读取未授权目录。
5. 保存长期记忆。
6. 调用外部敏感工具。
7. 导出用户数据。

## 8. Task Router 设计

Task Router 负责把用户输入转成可执行任务。

### 8.1 任务类型

| 类型 | 示例 |
|---|---|
| chat | 普通问答 |
| summarize_page | 总结当前网页 |
| analyze_error | 分析当前报错 |
| inspect_project | 分析项目目录 |
| write_file | 生成文件 |
| run_command | 执行命令 |
| update_memory | 记忆更新 |

### 8.2 路由规则

```text
用户输入
  -> 判断是否需要上下文
  -> 判断上下文类型
  -> 判断是否需要权限
  -> 生成任务
  -> 调用 Agent Runtime
  -> 返回状态事件流
```

### 8.3 任务生命周期

```text
created
context_collecting
planning
waiting_approval
running
memory_confirming
completed
failed
cancelled
```

## 9. 权限与安全设计

### 9.1 权限等级

| 等级 | 行为 | 是否确认 |
|---|---|---|
| L0 | 普通聊天 | 不需要 |
| L1 | 读取当前窗口标题 | 首次需要 |
| L2 | 读取屏幕或浏览器内容 | 首次需要，可撤销 |
| L3 | 读取授权文件 | 需要目录授权 |
| L4 | 写入文件 | 每次确认 |
| L5 | 执行命令 | 每次确认 |
| L6 | 删除、支付、发送、发布 | 初版禁止 |

### 9.2 命令执行确认

命令执行弹窗必须显示：

```text
AI 想执行以下命令：

命令：
npm test

目的：
验证项目测试是否通过

风险：
可能产生缓存或日志文件

[允许本次] [拒绝] [复制命令]
```

### 9.3 文件写入确认

文件写入弹窗必须显示：

```text
AI 想写入文件：

路径：
C:/project/report.md

写入原因：
保存当前网页总结结果

变更类型：
新建文件

[允许写入] [拒绝] [预览内容]
```

### 9.4 审计日志

所有敏感行为记录到本地日志：

1. 任务 ID。
2. 操作类型。
3. 操作目标。
4. 用户选择。
5. 时间。
6. 执行结果。

## 10. 记忆系统设计

### 10.1 记忆原则

1. 默认不自动写入长期记忆。
2. 每次写入长期记忆都必须弹窗确认。
3. 用户可以编辑、删除、导出记忆。
4. 记忆要区分用户偏好、项目事实、技能流程。
5. 敏感信息不自动保存。

### 10.2 记忆分类

| 类型 | 内容 |
|---|---|
| 用户记忆 | 用户偏好、工作习惯、常用工具 |
| 项目记忆 | 项目结构、技术栈、约定、目录 |
| 技能记忆 | 可复用操作流程 |
| 任务历史 | 已执行任务、结果和上下文摘要 |

### 10.3 记忆文件结构

```text
memory/
  user.md
  projects/
    project_id.md
  skills/
    summarize_page.md
    analyze_error.md
  history/
    task_20260610_001.json
```

### 10.4 记忆确认弹窗

```text
AI 认为这条信息可能有助于以后工作：

“用户偏好：生成方案时希望直接给可执行落地版本。”

是否保存到长期记忆？

[保存] [仅本次有效] [不保存]
```

### 15.1 产品验收

1. 用户可以看到桌面小人。
2. 用户可以通过快捷键唤起输入。

6. 小人能显示状态动画。

8. 任务结果可复制或保存。

### 15.2 安全验收

1. 未授权目录不可读取。
2. 写文件前必须确认。
3. 执行命令前必须确认。
4. 长期记忆保存前必须确认。
5. 用户可删除记忆。
6. 用户可撤销权限。
7. 所有高危操作有本地日志。

### 15.3 封测验收

1. 安装包可用。
2. 基础体验稳定。
3. 连续运行 8 小时无明显内存泄漏。
4. 三个核心场景成功率达到可演示水平。
5. 新用户 10 分钟内能理解产品用途。

## 17. 风险与处理

风险一：第三方许可证风险

处理方式：

1. 不直接复制第三方商业风险代码。
2. 对 Everywhere、Hermes、OpenHuman 等项目逐项复核许可证。
3. 商业版核心能力尽量自研或使用明确可商用组件。

风险二：用户不信任 AI 操作电脑

处理方式：

1. 所有高危操作必须确认。
2. 显示操作目的、目标和风险。
3. 提供权限中心和审计日志。

风险三：小人变成聊天框皮肤

处理方式：

1. 所有任务动作必须进入状态机。
2. 小人动画必须表达任务阶段。
3. 任务详情面板必须展示步骤、工具和进度。

风险四：功能过多导致做不完

处理方式：

1. 初版只围绕网页总结、报错分析、项目目录诊断三个场景。
2. 语音、Live2D、插件市场、多角色全部延后。
3. 每周都设置可验收成果。

## 18. 立即开始事项

建议按以下顺序启动：

1. 确定产品名和初版视觉风格。
2. 选择桌面技术路线：优先 C# / Avalonia，备选 Tauri。
3. 先做悬浮小人和状态机，不先做复杂 Agent。
4. 定义状态事件协议。
5. 实现任务面板。
6. 接入一个最小 Agent Runtime。
7. 做“总结当前网页”作为第一个端到端闭环。
8. 加入权限确认。
9. 加入手动记忆确认。
10. 打包第一个 Windows 封测版本。

## 19. 初版总结

初版成功的关键不是角色多精美，也不是功能多全，而是把“桌面小人 + 当前上下文 + Agent 工作流 + 状态动画 + 权限确认 + 用户可控记忆”这一条链路做通。只要用户能在桌面上唤起小人，让它理解当前屏幕并完成一个可见、可控、可追踪的任务，这个产品就具备封测价值。
